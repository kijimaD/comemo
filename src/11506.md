# [インデックス 11506] ファイルの概要

このコミットは、Go言語のコマンドラインツール（`cmd/go`）におけるテスト実行時の挙動を改善するものです。具体的には、テストが完了した際に生成される一時ディレクトリを即座にクリーンアップすることで、ディスク使用量を大幅に削減します。

Author: Joel Sing <jsing@google.com>
Date:   Tue Jan 31 10:37:21 2012 -0500

    A go build currently generates around 400MB of test output prior to
    cleaning up. With this change we use a maximum of ~15MB.

## 元コミット内容

`cmd/go: clean test directories as they complete`

このコミットは、Goのビルドプロセスにおいて、テストが完了した際にテスト用の一時ディレクトリをクリーンアップする機能を追加します。これにより、テスト実行中に生成される一時ファイルの総量を大幅に削減し、ディスクスペースの消費を抑えることを目的としています。変更前は最大400MBものテスト出力が生成されていましたが、この変更により最大約15MBにまで削減されると述べられています。

## 変更の背景

Go言語のビルドシステム、特に`go test`コマンドは、テスト実行時に一時的なディレクトリを作成し、そこにテストバイナリやその他の関連ファイルを配置します。これらのファイルはテストの実行に必要なものであり、テストが完了した後にクリーンアップされることが期待されます。しかし、このコミットが導入される以前は、テストが完了してもすぐに一時ディレクトリが削除されず、すべてのテストが終了するまで残ってしまうという問題がありました。

この挙動は、特に大規模なプロジェクトや、多数のテストパッケージを持つGoの標準ライブラリのような環境において、深刻なディスクスペースの消費を引き起こしていました。コミットメッセージに記載されているように、「`go build`はクリーンアップ前に約400MBものテスト出力を生成する」という状況は、開発者のディスク容量を圧迫し、CI/CD環境などでのビルド時間の増加や、ディスクI/Oの負荷増大にも繋がりかねません。

この問題を解決するため、テストが個別に完了するたびに、そのテストに関連する一時ディレクトリを即座にクリーンアップするメカニズムを導入する必要がありました。これにより、同時に存在する一時ファイルの総量を最小限に抑え、ディスク使用量を大幅に削減することが可能になります。コミットメッセージにある「この変更により、最大約15MBの使用量になる」という記述は、この改善の大きな効果を示しています。

## 前提知識の解説

このコミットを理解するためには、以下のGo言語および`cmd/go`に関する基本的な知識が必要です。

### Go言語のビルドシステム (`go build`, `go test`)

Go言語は、`go`コマンドを通じてビルド、テスト、実行、フォーマットなど、様々な開発タスクを統合的に管理します。
- **`go build`**: ソースコードをコンパイルして実行可能なバイナリを生成します。
- **`go test`**: パッケージ内のテストコードを実行します。`go test`は、テスト対象のパッケージとテストコードをコンパイルし、一時的な実行可能ファイルを生成して実行します。この際、テストバイナリやテスト中に生成される一時ファイルが特定のディレクトリに配置されます。

### `cmd/go`パッケージ

`cmd/go`は、Go言語の標準ツールチェーンの中核をなすパッケージであり、`go`コマンドの実装を含んでいます。このパッケージは、Goのソースコードの解析、依存関係の解決、コンパイル、リンク、テスト実行など、Go開発におけるほとんどの操作を司っています。`src/cmd/go/test.go`ファイルは、`go test`コマンドのロジック、特にテストのビルド、実行、結果の処理、そして一時ファイルの管理に関する部分を定義しています。

### `action`構造体とビルドグラフ

`cmd/go`内部では、ビルドやテストの各ステップが「アクション（`action`）」として表現され、依存関係を持つ有向非巡回グラフ（DAG）として管理されます。
- **`action`**: 特定のタスク（例: コンパイル、テスト実行、ファイルのクリーンアップ）を表す構造体です。
- **`deps`**: 各`action`は、実行前に完了していなければならない他の`action`への依存関係（`deps`）を持つことができます。これにより、Goのビルドシステムは、タスクの並列実行や正しい順序での実行を効率的に管理します。
- **`builder`**: ビルドプロセス全体を管理する構造体で、`action`の実行を調整します。

このコミットでは、この`action`メカニズムを利用して、テスト完了後のクリーンアップ処理をビルドグラフに組み込んでいます。

### 一時ディレクトリとテスト出力

`go test`が実行されると、テスト対象のパッケージごとに一時的な作業ディレクトリが作成されます。このディレクトリは通常、`$GOCACHE`の下や、`go build`が管理する一時的なビルドキャッシュ内に配置されます。テストバイナリ、テスト中に生成されるログファイル、カバレッジデータなどがこのディレクトリに保存されます。テストが完了すると、これらのファイルは通常削除されますが、このコミット以前は、その削除タイミングが遅れることが問題でした。

## 技術的詳細

このコミットの技術的な核心は、Goのビルドシステムがテスト実行を管理する方法に、テスト後の一時ディレクトリの即時クリーンアップを統合した点にあります。これは、`cmd/go`内部の`action`メカニズムを巧みに利用して実現されています。

### 問題点と解決策

**変更前の問題点**:
`go test`は、各テストパッケージに対して一時的な作業ディレクトリを作成し、テストバイナリやテスト中に生成されるファイルをそこに配置します。以前の挙動では、これらのディレクトリはテストが完了してもすぐに削除されず、すべてのテストが終了するまで残っていました。これにより、特に多数のテストパッケージがある場合、ディスク上に大量の一時ファイルが蓄積され、最大で400MBものディスクスペースを消費していました。

**解決策**:
このコミットは、テスト実行の`action`が完了した直後に、そのテストに関連する一時ディレクトリを削除する新しい`action` (`cleanTest`) を導入します。この`cleanTest`アクションは、既存の`printTest`アクション（テスト結果を標準出力に出力するアクション）の前に実行されるように依存関係が設定されます。これにより、テスト結果が出力される頃には、関連する一時ファイルは既に削除されている状態になります。

### `action`の依存関係の変更

Goのビルドシステムは、各タスクを`action`として扱い、これらの`action`間の依存関係を管理することで、実行順序と並列性を制御します。
変更前は、`printAction`（テスト結果の出力）が`runAction`（テストの実行）に直接依存していました。

```go
// 変更前
printAction = &action{
    f:    (*builder).printTest,
    deps: []*action{runAction}, // printActionはrunActionに直接依存
    p:    p,
}
```

このコミットでは、`runAction`と`printAction`の間に新しい`cleanAction`を挿入します。

1.  **`cleanAction`の導入**:
    `cleanTest`という新しい関数が`builder`のメソッドとして追加され、これが`cleanAction`の実行関数となります。この`cleanAction`は、`runAction`（テスト実行）が完了した後に実行されるように依存関係が設定されます。

    ```go
    cleanAction := &action{
        f:    (*builder).cleanTest, // 新しいクリーンアップ関数
        deps: []*action{runAction}, // runActionが完了したら実行
        p:    p,
    }
    ```

2.  **`printAction`の依存関係の変更**:
    `printAction`は、もはや`runAction`に直接依存せず、新しく導入された`cleanAction`に依存するように変更されます。これにより、`printAction`が実行される（つまりテスト結果が出力される）前に、必ず`cleanAction`が実行され、一時ディレクトリがクリーンアップされることが保証されます。

    ```go
    printAction = &action{
        f:    (*builder).printTest,
        deps: []*action{cleanAction}, // cleanActionが完了したら実行
        p:    p,
    }
    ```

この依存関係の連鎖により、`runAction` -> `cleanAction` -> `printAction`という実行順序が確立され、テストが完了するたびにそのテストに関連する一時ディレクトリが即座に削除されるようになります。

### `cleanTest`関数の実装

`cleanTest`関数は、`runAction`が生成した一時ディレクトリのパスを特定し、`os.RemoveAll`関数を使用してそのディレクトリを再帰的に削除します。一時ディレクトリのパスは、`builder.work`（ビルドの作業ディレクトリ）とパッケージのインポートパス（`run.p.ImportPath`）から構築されます。

```go
// cleanTest is the action for cleaning up after a test.
func (b *builder) cleanTest(a *action) error {
    run := a.deps[0] // cleanActionの依存はrunAction
    // テストディレクトリのパスを構築
    testDir := filepath.Join(b.work, filepath.FromSlash(run.p.ImportPath+"/_test"))
    os.RemoveAll(testDir) // ディレクトリを削除
    return nil
}
```

この変更により、Goのテスト実行プロセスは、ディスクスペースの効率的な利用という点で大幅に改善されました。

### 3. 新しい`cleanTest`関数の追加

このコミットの主要な機能追加です。

```go
+// cleanTest is the action for cleaning up after a test.
+func (b *builder) cleanTest(a *action) error {\n+\trun := a.deps[0]\n+\ttestDir := filepath.Join(b.work, filepath.FromSlash(run.p.ImportPath+\"/_test\"))\n+\tos.RemoveAll(testDir)\n \treturn nil\n }\n```

- `// cleanTest is the action for cleaning up after a test.`: 関数の目的を説明するコメントです。
- `func (b *builder) cleanTest(a *action) error {`: `builder`型に`cleanTest`というメソッドが追加されました。これは`action`の実行関数として使用されます。
- `run := a.deps[0]`: `cleanAction`は`runAction`に依存しているため、`a.deps[0]`は`runAction`を指します。ここから、テストが実行されたパッケージの情報（`run.p`）を取得します。
- `testDir := filepath.Join(b.work, filepath.FromSlash(run.p.ImportPath+"/_test"))`:
    - `b.work`: `builder`の作業ディレクトリ（一時ファイルが生成されるルートディレクトリ）。
    - `run.p.ImportPath`: テストが実行されたパッケージのインポートパス（例: "fmt", "net/http"）。
    - `filepath.FromSlash(...)`: OS固有のパス区切り文字に変換します。
    - `"/_test"`: Goのテストシステムが一時ファイルを配置する慣例的なサブディレクトリ名です。
    - これらを結合することで、特定のテストパッケージのための一時ディレクトリの絶対パスが構築されます。
- `os.RemoveAll(testDir)`: 構築された`testDir`パスにあるディレクトリとその内容をすべて再帰的に削除します。これにより、テストが完了した直後に一時ファイルがクリーンアップされます。
- `return nil`: エラーなくクリーンアップが完了したことを示します。

### 4. `printTest`関数の変更点

`printTest`関数は、テスト結果を標準出力に書き出す`action`の実行関数です。

```go

- **依存関係の参照の変更**:
    - `-	run := a.deps[0]`: 変更前は、`printAction`が`runAction`に直接依存していたため、`a.deps[0]`は`runAction`を指していました。
    - `+	clean := a.deps[0]`: 変更後、`printAction`は`cleanAction`に依存するようになったため、`a.deps[0]`は`cleanAction`を指します。
    - `+	run := clean.deps[0]`: `cleanAction`は`runAction`に依存しているため、`clean.deps[0]`を通じて元の`runAction`にアクセスし、その`testOutput`（テスト結果のバッファ）を取得します。
- `os.Stdout.Write(run.testOutput.Bytes())`: `runAction`から取得したテスト結果のバイト列を標準出力に書き出します。
- `run.testOutput = nil`: テスト出力バッファを解放し、メモリをクリーンアップします。
