`log/syslog: fix documentation for NewLogger`

このコミットは、`log/syslog`パッケージの`NewLogger`関数のドキュメントを修正し、Issue #2798を解決することを目的としています。

## 変更の背景

この変更の背景には、Go 1のリリースに向けたAPIの安定化と改善があります。特に、`log/syslog.NewLogger`関数が、Syslogへの接続確立に失敗した場合にエラーを適切に通知できるようにする必要がありました。元の実装では、エラーが発生しても`nil`を返すだけで、呼び出し元がエラーの原因を特定したり、適切にハンドリングしたりすることが困難でした。

Issue #2798は、この`NewLogger`関数のエラーハンドリングの欠陥を指摘しており、関数が`log.Logger`だけでなくエラーも返すように変更することで、より堅牢なエラー処理を可能にすることが求められました。これにより、アプリケーションはSyslogへの接続失敗を検知し、適切なフォールバック処理やエラーロギングを行うことができるようになります。

また、Go 1のリリースでは、既存のAPIに対する後方互換性のない変更（breaking changes）が導入される場合があり、その際には`go1.html`や`go1.tmpl`といったドキュメントでその変更内容と、既存コードを更新するためのガイダンスが提供されました。このコミットも、`NewLogger`関数のシグネチャ変更というbreaking changeを反映するために、これらのドキュメントを更新しています。

## 前提知識の解説

### Go言語の`log`パッケージと`log/syslog`パッケージ

*   **`log`パッケージ**: Go言語の標準ライブラリに含まれる基本的なロギング機能を提供するパッケージです。`log.Logger`型を通じて、メッセージの出力先（`io.Writer`インターフェースを満たすもの）、プレフィックス、ロギングフラグ（タイムスタンプ、ファイル名など）を設定できます。
*   **`log/syslog`パッケージ**: `log`パッケージと連携し、ログメッセージをUnix系システムで標準的に利用されるSyslogサービスに送信するための機能を提供します。これにより、システム全体のログ管理システムと統合された形でアプリケーションのログを記録できます。

### Syslog

Syslogは、システムメッセージやイベントログを収集するための標準的なプロトコルです。通常、デーモンプロセスやアプリケーションはSyslogプロトコルを使用してログメッセージをSyslogデーモン（例: `syslogd`, `rsyslogd`, `journald`など）に送信し、Syslogデーモンはそれらのメッセージを一元的に管理、保存、または転送します。Syslogメッセージには、ファシリティ（メッセージの発生源、例: `kern`, `user`, `mail`など）とプライオリティ（メッセージの重要度、例: `emerg`, `alert`, `crit`, `err`, `warning`, `notice`, `info`, `debug`など）が含まれます。

### `go fix`コマンド

`go fix`コマンドは、Go言語のツールチェーンの一部であり、Goのバージョンアップに伴うAPIの変更や非推奨になった機能に対して、既存のGoソースコードを自動的に更新するツールです。しかし、すべての変更に対応できるわけではなく、特にシグネチャの変更など、より複雑な変更は手動での修正が必要となる場合があります。

### `interface{}`と`reflect.Value`

Go言語における型表現の概念です。
*   **`interface{}`**: 任意の型の値を保持できる空のインターフェースです。Goのジェネリクスが導入される前は、様々な型の値を扱うための一般的な方法でした。
*   **`reflect.Value`**: `reflect`パッケージの一部で、Goの実行時リフレクション機能を提供します。これにより、プログラムは実行時に変数や関数の型情報を検査したり、値を操作したりできます。`reflect.Value`は、特定のGoの値を抽象的に表現するために使用されます。

このコミットでは、`encoding/binary.Size`関数が`reflect.Value`から`interface{}`を引数に取るように変更されたことにも言及されており、これはAPIの使いやすさの改善と、リフレクションの使用を減らすことによるパフォーマンス向上を意図している可能性があります。

## 技術的詳細

このコミットの主要な技術的変更は、`log/syslog`パッケージの`NewLogger`関数のシグネチャ変更です。

**変更前:**
`func NewLogger(p Priority, flag int) *log.Logger`

**変更後:**
`func NewLogger(p Priority, logFlag int) (*log.Logger, error)`

この変更により、`NewLogger`関数は`log.Logger`のポインタだけでなく、`error`型も返すようになりました。これは、`New`関数（`syslog.New`）がSyslogへの接続確立に失敗した場合にエラーを返す可能性があるため、そのエラーを呼び出し元に伝播させるための重要な変更です。

具体的には、`syslog.New`関数がエラーを返した場合、以前は`NewLogger`関数内で`nil`を返していましたが、変更後はそのエラーをそのまま`NewLogger`の呼び出し元に返すようになりました。これにより、`NewLogger`の呼び出し元は、Syslogロガーの初期化が成功したかどうかを明示的にチェックし、エラーが発生した場合には適切なエラーハンドリングを行うことができます。

また、この変更は`go fix`コマンドでは自動的に修正できない種類の変更であるため、`go1.html`および`go1.tmpl`のドキュメントに、この変更がコンパイラによって捕捉され、手動での更新が必要である旨が追記されています。これは、Go言語のAPI変更における互換性維持のポリシーと、開発者への適切な情報提供の重要性を示しています。

## コアとなるコードの解説

### `src/pkg/log/syslog/syslog.go` の変更

1.  **関数のシグネチャ変更**:
    *   変更前: `func NewLogger(p Priority, flag int) *log.Logger`
    *   変更後: `func NewLogger(p Priority, logFlag int) (*log.Logger, error)`
    `NewLogger`関数は、`*log.Logger`だけでなく、`error`も返すようになりました。引数名も`flag`から`logFlag`に変更され、より意図が明確になっています。

2.  **エラーハンドリングの追加**:
    *   `s, err := New(p, "")` で`syslog.New`関数を呼び出し、Syslogへの接続を試みます。
    *   `if err != nil { return nil, err }` の行が追加されました。これにより、`syslog.New`がエラーを返した場合、そのエラーを`NewLogger`の呼び出し元に直接伝播させます。以前は`nil`を返していただけでした。
    *   成功した場合の戻り値も `return log.New(s, "", logFlag), nil` となり、エラーが`nil`であることを明示的に示しています。

この変更により、`NewLogger`の呼び出し元は、Syslogロガーの初期化が成功したかどうかを確実に判断できるようになり、エラー発生時には適切なフォールバック処理を実装できます。

### `src/pkg/log/syslog/syslog_test.go` の変更

1.  **テストコードの更新**:
    *   変更前: `f := NewLogger(LOG_INFO, 0)`
    *   変更後: `f, err := NewLogger(LOG_INFO, 0)`
    `NewLogger`がエラーを返すようになったため、テストコードもその戻り値を受け取るように変更されています。

2.  **エラーチェックの改善**:
    *   変更前: `if f == nil { t.Error("NewLogger() failed") }`
    *   変更後: `if f == nil { t.Error(err) }`
    `NewLogger`が`nil`を返した場合、以前は一般的なエラーメッセージを出力していましたが、変更後は`NewLogger`が返した実際のエラーオブジェクトを`t.Error`に渡すようになりました。これにより、テスト失敗時のデバッグ情報がより詳細になります。

これらの変更は、`NewLogger`関数のAPI変更にテストコードを適応させ、エラーハンドリングのテストカバレッジを向上させるものです。
