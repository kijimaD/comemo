# [インデックス 10801] ファイルの概要

このコミットでは、Go言語の標準ライブラリである`fmt`パッケージの動作を修正しています。具体的には、`fmt`パッケージが値の文字列化を行う際に、その値が`String()`メソッド（`fmt.Stringer`インターフェースを実装している場合）を呼び出すことで発生する可能性のある無限再帰の問題に対処しています。

変更されたファイルは以下の2つです。

*   `src/pkg/fmt/fmt_test.go`: このファイルには、`fmt`パッケージのテストコードが含まれています。今回のコミットでは、`String()`メソッドが誤動作した場合に無限再帰が発生しないことを確認するための新しいテストケースが追加されています。
*   `src/pkg/fmt/print.go`: このファイルは、`fmt`パッケージの主要な印刷ロジックを実装しています。今回のコミットでは、再帰を防ぐためのロジックが追加されています。

## 変更の背景

このコミットは、Go言語の`fmt`パッケージにおける特定のバグ、Issue 2555を修正するために行われました。この問題は、ユーザー定義型が`fmt.Stringer`インターフェース（`String() string`メソッドを持つ）を実装している場合に発生する可能性がありました。

具体的には、`fmt`パッケージが値を文字列化する際に、その値が`String()`メソッドを持っていると、`fmt`は自動的にそのメソッドを呼び出して文字列表現を取得します。しかし、もしこの`String()`メソッドの内部で、再び`fmt`パッケージの関数（例: `fmt.Sprintf`）を使って自身のインスタンスをフォーマットしようとすると、`fmt`は再度`String()`メソッドを呼び出そうとし、結果として無限再帰に陥り、最終的にはスタックオーバーフローを引き起こす可能性がありました。

このコミットの目的は、このような「誤動作する`String()`メソッド」が引き起こす無限再帰を防ぎ、`fmt`パッケージの堅牢性を向上させることでした。`fmt`パッケージがエラー状態にある間は、ユーザー定義の`String()`メソッドの呼び出しを抑制することで、この問題が解決されています。

## 前提知識の解説

### `fmt`パッケージと文字列フォーマット

Go言語の`fmt`パッケージは、C言語の`printf`に似た書式設定されたI/O機能を提供します。`fmt.Print`, `fmt.Println`, `fmt.Sprintf`などの関数を通じて、様々な型の値を文字列に変換して出力したり、文字列として取得したりすることができます。

### `fmt.Stringer`インターフェース

`fmt.Stringer`はGoの標準ライブラリで定義されているインターフェースの一つです。

```go
type Stringer interface {
    String() string
}
```

このインターフェースを実装する型は、`String()`という名前のメソッドを持ち、そのメソッドは自身の値を人間が読める文字列形式で返す責任があります。`fmt`パッケージの関数は、値をフォーマットする際に、その値が`fmt.Stringer`インターフェースを実装しているかどうかをチェックします。もし実装していれば、`fmt`はデフォルトでその`String()`メソッドを呼び出し、返された文字列をフォーマット結果として使用します。これにより、開発者は独自の型を`fmt`パッケージで自然に表示させることができます。

### 無限再帰とスタックオーバーフロー

再帰とは、関数が自分自身を呼び出すプログラミングのテクニックです。無限再帰は、再帰関数が終了条件を満たさずに自分自身を無限に呼び出し続ける状態を指します。

コンピュータのプログラムは、関数呼び出しの情報をスタックと呼ばれるメモリ領域に保存します。関数が呼び出されるたびに、その関数のローカル変数や戻りアドレスなどの情報がスタックに積まれます。無限再帰が発生すると、スタックに情報が際限なく積まれ続け、最終的にスタック領域が枯渇します。この状態を「スタックオーバーフロー」と呼び、プログラムは通常、クラッシュして終了します。

今回の問題では、`fmt.Stringer`の`String()`メソッドが`fmt`関数を呼び出し、その`fmt`関数が再び`String()`メソッドを呼び出すという循環参照が発生し、無限再帰とスタックオーバーフローを引き起こしていました。

## 技術的詳細

このコミットの技術的な核心は、`fmt`パッケージが「エラー状態」にある間は、ユーザー定義の`String()`メソッドの呼び出しを抑制するという点にあります。

`fmt`パッケージの内部では、`pp`という構造体がフォーマット処理の状態を管理しています。このコミットでは、`pp`構造体に`erroring`という新しいブール型フィールドが追加されました。

*   `erroring bool`: このフラグは、`fmt`パッケージが現在エラー状態（例えば、不正な動詞が指定された場合など）でフォーマット処理を行っているかどうかを示します。

変更のロジックは以下の通りです。

1.  **`newPrinter()`での初期化**: `pp`構造体のインスタンスが新しく作成される際（`newPrinter()`関数内）に、`erroring`フラグは`false`に初期化されます。これは、通常の状態ではエラー状態ではないことを意味します。
2.  **`badVerb()`での設定**: `fmt`パッケージが不正なフォーマット動詞（例: `%!p(value)`のようなエラーメッセージ）を処理する際に呼び出される`badVerb()`関数内で、`erroring`フラグが`true`に設定されます。これは、`fmt`が現在、ユーザーの入力に起因するエラーメッセージを生成している状態であることを示します。
3.  **`badVerb()`でのリセット**: `badVerb()`関数がエラーメッセージの生成を完了すると、`erroring`フラグは`false`に戻されます。
4.  **`handleMethods()`での抑制**: `handleMethods()`関数は、`fmt`パッケージが値のフォーマットを行う際に、その値が`fmt.Formatter`や`fmt.Stringer`などのインターフェースを実装しているかどうかをチェックし、対応するメソッドを呼び出す役割を担っています。この関数に、`if p.erroring { return }`というガード句が追加されました。
    このガード句により、もし`fmt`パッケージが現在エラー状態（`p.erroring`が`true`）であれば、`fmt.Formatter`や`fmt.Stringer`などのユーザー定義メソッドの呼び出しがスキップされます。

このメカニズムにより、`String()`メソッドが内部で`fmt.Sprintf`を呼び出し、その`fmt.Sprintf`が不正な動詞などでエラー状態になったとしても、`fmt`パッケージは再帰的に`String()`メソッドを呼び出すことを防ぎます。これにより、無限再帰とスタックオーバーフローが回避されます。

## コアとなるコードの解説

### `src/pkg/fmt/fmt_test.go` の変更点

このファイルには、`TestBadVerbRecursion`という新しいテスト関数が追加されています。このテストは、`String()`メソッドが誤動作した場合に`fmt`パッケージが無限再帰に陥らないことを検証します。

*   `Recur`構造体: テストのために定義されたカスタム型です。
    *   `i int`: 内部の整数値。
    *   `failed *bool`: `String()`メソッドが無限再帰に陥った場合に`true`に設定されるフラグへのポインタ。
*   `recurCount`: グローバル変数で、`Recur.String()`メソッドが呼び出された回数をカウントします。
*   `Recur.String() string`メソッド:
    *   このメソッドは、`recurCount`が10を超えると`*r.failed = true`を設定し、"FAIL"を返します。これは、`String()`メソッドが期待通りに動作しない（無限再帰に陥る）状況をシミュレートしています。
    *   重要なのは、`Sprintf("recur@%p value: %d", r, r.i)`という行です。ここで`r`（`Recur`型の値）を`%p`動詞でフォーマットしようとしています。`%p`はポインタのアドレスを表示するための動詞ですが、値型に対して使用すると、`fmt`パッケージは内部的にエラー状態となり、`%!p(value)`のようなエラーメッセージを生成しようとします。
    *   修正前は、このエラーメッセージの生成過程で、`fmt`は再度`Recur.String()`を呼び出そうとし、無限再帰が発生していました。修正後は、`fmt`がエラー状態にある間は`String()`メソッドの呼び出しが抑制されるため、再帰が停止します。
*   `TestBadVerbRecursion`関数:
    *   `Recur`型のインスタンスを作成し、`Sprintf`を呼び出してテストを行います。
    *   ポインタと値の両方でテストを行い、`failed`フラグが`true`になっていないことを確認します。もし`failed`が`true`になっていれば、それは無限再帰が発生したことを意味し、テストは失敗します。

### `src/pkg/fmt/print.go` の変更点

このファイルは`fmt`パッケージの内部実装であり、主に`pp`構造体とそのメソッドがフォーマット処理を担当しています。

*   `pp`構造体への`erroring`フィールドの追加:
    ```go
    type pp struct {
        n         int
        panicking bool
        erroring  bool // printing an error condition
        buf       bytes.Buffer
        // ...
    }
    ```
    `erroring`という新しいブール型フィールドが追加されました。これは、`pp`インスタンスが現在エラー条件を処理中であるかどうかを示すフラグです。

*   `newPrinter()`での`erroring`の初期化:
    ```go
    func newPrinter() *pp {
        p := ppFree.get().(*pp)
        p.panicking = false
        p.erroring = false // ここでfalseに初期化
        p.fmt.init(&p.buf)
        return p
    }
    ```
    新しい`pp`インスタンスがプールから取得される際に、`erroring`フラグが`false`に初期化されます。

*   `badVerb()`での`erroring`の設定とリセット:
    ```go
    func (p *pp) badVerb(verb rune) {
        p.erroring = true // エラー処理開始時にtrueに設定
        p.add('%')
        p.add('!')
        p.add(verb)
        // ...
        p.add(')')
        p.erroring = false // エラー処理終了時にfalseに戻す
    }
    ```
    `badVerb()`関数は、`fmt`パッケージが不正なフォーマット動詞に遭遇した際に呼び出されます。この関数が実行されている間、`erroring`フラグが`true`に設定され、エラーメッセージの生成が完了すると`false`に戻されます。

*   `handleMethods()`での`erroring`によるガード:
    ```go
    func (p *pp) handleMethods(verb rune, plus, goSyntax bool, depth int) (wasString, handled bool) {
        if p.erroring { // erroringがtrueの場合、ここで処理を中断
            return
        }
        // Is it a Formatter?
        if formatter, ok := p.field.(Formatter); ok {
            handled = true
            // ...
        }
        // ...
    }
    ```
    `handleMethods()`関数は、`fmt.Stringer`や`fmt.Formatter`などのインターフェースを実装している型に対して、対応するメソッドを呼び出す役割を担っています。この関数の冒頭に`if p.erroring { return }`というガード句が追加されました。これにより、`fmt`がエラー状態にある間は、ユーザー定義の`String()`メソッド（および`Formatter`メソッド）の呼び出しがスキップされ、無限再帰が防止されます。

これらの変更により、`fmt`パッケージは、`String()`メソッドが予期せぬ動作（例えば、自身を再帰的にフォーマットしようとする）をした場合でも、安全にエラーを処理し、スタックオーバーフローを防ぐことができるようになりました。
