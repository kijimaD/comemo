## 元コミット内容

このコミットは、Go言語の標準ライブラリ`os`パッケージに`ModeSticky`という新しいファイルモード定数を追加し、Unix系システムにおけるスティッキービットのサポートを導入するものです。これにより、Goプログラムがファイルのパーミッション情報を扱う際に、スティッキービットの状態を正確に認識し、操作できるようになります。

具体的には、以下の変更が含まれています。

*   `src/pkg/os/types.go`:
    *   `ModeSticky`という`FileMode`定数を追加。
    *   `FileMode`の文字列表現（`String()`メソッド）に、スティッキービットを示す`t`を追加。
    *   `ModeTemporary`のコメント表記を`t:`から`T:`に変更し、`ModeSticky`との混同を避ける。
*   `src/pkg/os/stat_darwin.go`, `src/pkg/os/stat_freebsd.go`, `src/pkg/os/stat_linux.go`, `src/pkg/os/stat_netbsd.go`, `src/pkg/os/stat_openbsd.go`:
    *   各OS固有の`fileInfoFromStat`関数において、`syscall.Stat_t`構造体のモード情報から`syscall.S_ISVTX`（スティッキービット）が設定されている場合に、新しく追加された`ModeSticky`フラグを`fs.mode`に設定するロジックを追加。

## 変更の背景

Go言語の`os`パッケージは、ファイルシステム操作のためのクロスプラットフォームなインターフェースを提供します。ファイルやディレクトリのパーミッションは、その重要な側面の一つです。従来の`os`パッケージでは、Unix系システムに存在する特殊なパーミッションビットである「スティッキービット」を直接的に表現する`FileMode`定数が存在しませんでした。

スティッキービットは、特に共有ディレクトリにおいて、そのディレクトリ内のファイルに対する削除・名前変更の権限を制御するために重要な役割を果たします。例えば、`/tmp`ディレクトリのように複数のユーザーが書き込み権限を持つ場所では、他のユーザーが作成したファイルを勝手に削除できないように、スティッキービットが設定されているのが一般的です。

このコミットは、Goプログラムがファイルやディレクトリのパーミッションをより正確に、かつ詳細に扱うことができるようにするために、スティッキービットのサポートを`os`パッケージに追加することを目的としています。これにより、Goで書かれたツールやアプリケーションが、Unix系システムのファイルシステムセマンティクスをより忠実に反映できるようになります。

## 前提知識の解説

### Unix系ファイルパーミッションと特殊ビット

Unix系オペレーティングシステムでは、ファイルやディレクトリには所有者、グループ、その他のユーザーに対する読み取り（r）、書き込み（w）、実行（x）の3種類のパーミッションが設定されます。これらは通常、`rwxrwxrwx`のような形式で表現されます。

これらに加えて、ファイルやディレクトリには「特殊パーミッションビット」と呼ばれる3つの追加のビットが存在します。

1.  **SetUID (Set User ID)**:
    *   実行可能ファイルに設定された場合、そのファイルを実行したユーザーのIDではなく、ファイルの所有者のIDでプロセスが実行されます。これにより、通常のユーザーが特定の特権操作（例: パスワード変更）を実行できるようになります。
    *   `ls -l`の出力では、所有者の実行権限（x）の位置が`s`に変わります（例: `rwsr-xr-x`）。
2.  **SetGID (Set Group ID)**:
    *   実行可能ファイルに設定された場合、そのファイルを実行したユーザーのプライマリグループではなく、ファイルのグループのIDでプロセスが実行されます。
    *   ディレクトリに設定された場合、そのディレクトリ内に新しく作成されるファイルやサブディレクトリは、親ディレクトリのグループを継承します。
    *   `ls -l`の出力では、グループの実行権限（x）の位置が`s`に変わります（例: `rwxrwsr-x`）。
3.  **スティッキービット (Sticky Bit)**:
    *   **ディレクトリに設定された場合**: そのディレクトリ内のファイルは、ファイルの所有者、ディレクトリの所有者、またはスーパーユーザー（root）のみが削除または名前変更できます。たとえディレクトリに対する書き込み権限を持っていても、他のユーザーが作成したファイルを勝手に削除することはできません。これは共有ディレクトリ（例: `/tmp`）で非常に重要です。
    *   **実行可能ファイルに設定された場合**: 過去には、プログラムのテキストセグメント（コード部分）をスワップ領域に保持し、再実行時のロードを高速化するために使用されました。しかし、現代のUnix系システムでは仮想メモリ管理の進化により、この機能はほとんど使われず、通常は効果がありません。
    *   `ls -l`の出力では、その他のユーザーの実行権限（x）の位置が`t`に変わります（例: `drwxrwxrwt`）。もしその他のユーザーの実行権限が設定されていない場合は`T`と表示されます（例: `drwxrwxrwT`）。

### `syscall.S_ISVTX`

`syscall.S_ISVTX`は、Unix系システムコールで使用される定数で、ファイルのモード情報（`stat`構造体などで取得される）からスティッキービットが設定されているかどうかをチェックするためのビットマスクです。`st.Mode & syscall.S_ISVTX`のようなビット演算を行うことで、スティッキービットの状態を判別できます。

### Go言語の`os.FileMode`

Go言語の`os`パッケージには、ファイルやディレクトリのパーミッションと種類を表す`FileMode`型があります。これはビットフィールドであり、様々なフラグを組み合わせてファイルの属性を表現します。

*   `os.ModeDir`: ディレクトリ
*   `os.ModeSymlink`: シンボリックリンク
*   `os.ModeSetuid`: SetUIDビット
*   `os.ModeSetgid`: SetGIDビット
*   など

これらのフラグはビット演算（`|`で結合、`&`でチェック）によって操作されます。`FileMode`型には`String()`メソッドが実装されており、`ls -l`コマンドの出力に似た形式でパーミッション情報を文字列として取得できます。

## 技術的詳細

このコミットの技術的詳細は、Go言語の`os`パッケージがどのようにUnix系システムのファイルパーミッション、特にスティッキービットを内部的に処理し、外部に公開するかという点に集約されます。

1.  **`FileMode`への`ModeSticky`の追加**:
    *   `src/pkg/os/types.go`において、`ModeSticky`という新しい`FileMode`定数が定義されました。これは、`FileMode`ビットフィールド内でスティッキービットを表す特定のビット位置を割り当てます。
    *   `const str = "dalTLDpSugct"`の変更は、`FileMode`の`String()`メソッドがパーミッションを文字列として表現する際に、スティッキービットが設定されている場合に`t`（または`T`）を表示するためのものです。`ModeTemporary`の表示文字が`t`から`T`に変更されたのは、`ModeSticky`と`ModeTemporary`が異なる概念であり、`t`がスティッキービットに割り当てられたため、衝突を避けるためです。

2.  **OS固有の`stat`システムコールとの連携**:
    *   Unix系OS（Darwin, FreeBSD, Linux, NetBSD, OpenBSD）のそれぞれに対応する`src/pkg/os/stat_*.go`ファイルが変更されました。
    *   これらのファイルには、`fileInfoFromStat`という関数があります。この関数は、OSの`syscall.Stat_t`構造体（`stat`システムコールによって取得されるファイル情報）を受け取り、Goの`os.FileInfo`インターフェースを実装する内部構造体（`fileStat`など）に変換する役割を担います。
    *   追加されたコード`if st.Mode&syscall.S_ISVTX != 0 { fs.mode |= ModeSticky }`は、`syscall.Stat_t`から取得したファイルモード（`st.Mode`）に対してビットマスク`syscall.S_ISVTX`を適用し、スティッキービットが立っているかどうかをチェックします。
    *   もしスティッキービットが立っていれば、Goの`FileMode`に対応する`fs.mode`に新しく定義された`ModeSticky`フラグをビットOR演算（`|=`）で設定します。これにより、OSから取得したスティッキービットの情報がGoの`FileMode`オブジェクトに正しくマッピングされるようになります。

この変更により、Goプログラムは`os.Stat()`や`os.Lstat()`などの関数を通じてファイル情報を取得した際に、返される`os.FileInfo`オブジェクトの`Mode()`メソッドから`os.ModeSticky`フラグをチェックすることで、そのファイルやディレクトリにスティッキービットが設定されているかどうかをプログラム的に判別できるようになります。

## コアとなるコードの解説

### `src/pkg/os/types.go`の変更

*   **`ModeSticky`定数の追加**:
    ```go
    ModeSticky FileMode = 1 << (32 - 1 - iota) // t: sticky
    ```
    `iota`はGoの定数宣言で使われる特殊な識別子で、連続する定数に自動的に値を割り当てます。この行は、`FileMode`型に`ModeSticky`という新しいビットフラグを導入しています。コメントの`t: sticky`は、`FileMode.String()`メソッドでこのフラグが`t`として表示されることを示唆しています。

*   **`FileMode.String()`メソッドの`str`定数変更**:
    ```diff
    -	const str = "daltLDpSugc"
    +	const str = "dalTLDpSugct"
    ```
    `FileMode`の`String()`メソッドは、`ls -l`コマンドの出力に似た形式でファイルモードを文字列に変換します。この`str`定数は、各ビットフラグに対応する文字を定義しています。末尾に`t`が追加されたことで、`ModeSticky`が設定されている場合に、出力文字列の適切な位置に`t`が表示されるようになります。

*   **`ModeTemporary`のコメント変更**:
    ```diff
    -	ModeTemporary                                  // t: temporary file (not backed up)
    +	ModeTemporary                                  // T: temporary file (not backed up)
    ```
    これは、`ModeSticky`が`t`を使用するため、`ModeTemporary`のコメント表記を`T`に変更し、視覚的な混同を避けるための調整です。実際の`FileMode`のビット値や動作には影響しません。

### `src/pkg/os/stat_*.go`の変更

```go
    if st.Mode&syscall.S_ISVTX != 0 {
        fs.mode |= ModeSticky
    }
```
このコードスニペットは、各OS固有の`fileInfoFromStat`関数内に挿入されています。
1.  `st.Mode`: これは`syscall.Stat_t`構造体から取得された、ファイルまたはディレクトリのパーミッションとタイプ情報を含む整数値です。これはOSが提供する生のモード情報です。
2.  `syscall.S_ISVTX`: これはUnix系システムコールで定義されている定数で、スティッキービットを表すビットマスクです。
3.  `st.Mode & syscall.S_ISVTX`: ビットAND演算子（`&`）を使用して、`st.Mode`に`syscall.S_ISVTX`で示されるビットが設定されているかどうかをチェックします。結果が非ゼロであれば、スティッキービットが設定されていることを意味します。
4.  `fs.mode |= ModeSticky`: もしスティッキービットが設定されていれば、Goの`os.FileMode`型である`fs.mode`に、新しく定義された`ModeSticky`フラグをビットOR演算子（`|=`）で追加します。これにより、OSのスティッキービット情報がGoの`FileMode`オブジェクトに正しく反映されます。

この変更により、Goプログラムは`os.Stat()`や`os.Lstat()`などの関数を呼び出してファイル情報を取得した際に、返される`os.FileInfo`インターフェースを通じてスティッキービットの状態を正確に取得し、それに基づいてロジックを組むことが可能になります。例えば、共有ディレクトリの挙動を模倣したり、特定のパーミッションを持つファイルを識別したりする際に役立ちます。
