# [インデックス 1324] ファイルの概要

このコミットは、Go言語の初期のビルドシステムにおける、Perforceからの新規チェックアウト時に発生する競合状態（race condition）を修正するものです。具体的には、`src/cmd/gc/Makefile`内の`sysimport.c`のビルドルールが変更され、`6g`コンパイラがまだ利用できない状況でもビルドが正常に進行するように改善されています。

## 変更の背景

この変更は、Go言語のソースコードをPerforceバージョン管理システムから新規にチェックアウトした際に発生するビルドエラーを解決するために導入されました。問題の根源は、`sysimport.c`というファイルが、その依存関係である`sys.go`、`unsafe.go`、`mksys.c`よりも古いタイムスタンプを持つ状態でチェックアウトされる可能性があったことにあります。

通常の`make`の動作では、ターゲットファイル（`sysimport.c`）のタイムスタンプが依存ファイルよりも古い場合、そのターゲットを再ビルドしようとします。しかし、Goのビルドプロセスはブートストラップ（自己コンパイル）の性質を持っており、初期段階ではGoコンパイラである`6g`がまだビルドされていない、あるいはパスが通っていない状態であることがあります。

このような状況で`sysimport.c`のビルドがトリガーされると、`6g`コマンドが見つからずにビルドが失敗し、Goのソースコード全体のコンパイルが中断されてしまうという競合状態が発生していました。このコミットは、`6g`が利用可能になる前に`sysimport.c`の不必要な再ビルドを回避し、ビルドプロセスがスムーズに進行できるようにするためのものです。

## 前提知識の解説

*   **Go言語の初期ビルドシステム**: Go言語は、そのコンパイラ自体がGoで書かれているため、ブートストラッププロセスが必要です。初期のGoコンパイラはC言語で書かれた`6c`, `6g`, `6l`などのツールチェーンで構成されており、これらを使ってGoのソースコードをコンパイルしていました。`6g`はGoソースコードをコンパイルするコンパイラです。
*   **`Makefile`**: `make`ユーティリティが使用するビルド設定ファイルです。依存関係とそれらを解決するためのコマンドを定義します。`make`はファイルのタイムスタンプを比較して、依存関係が変更された場合にのみターゲットを再ビルドします。
*   **`sysimport.c`**: Go言語のランタイムや標準ライブラリの一部をC言語から利用するためのインターフェースを提供するファイルの一つです。Goの内部構造やシステムコールへのアクセスをC言語側から行うために、`mksys`というツールによって生成されます。
*   **`mksys`**: `sys.go`や`unsafe.go`などのGoソースファイルから、C言語でGoの内部構造や関数を呼び出すためのCヘッダファイルやCソースファイルを生成するツールです。このツール自体はC言語で書かれており、`gcc`でコンパイルされます。
*   **`6g`**: 初期Go言語のコンパイラの一つで、Goのソースコードをコンパイルしてオブジェクトファイルを生成します。
*   **Perforce**: 集中型バージョン管理システム（Centralized Version Control System, CVCS）の一つです。Gitのような分散型システムとは異なり、すべての変更履歴が中央サーバーに保存されます。新規チェックアウト時には、サーバーから最新のファイルがローカルに取得されますが、この際にファイルのタイムスタンプがサーバー上のタイムスタンプで設定されることがあります。
*   **競合状態 (Race Condition)**: 複数のプロセスやスレッドが共有リソースにアクセスする際に、その実行順序によって結果が変わってしまう状態を指します。このケースでは、`make`が`sysimport.c`の再ビルドを試みるタイミングと、`6g`コンパイラが利用可能になるタイミングとの間に競合がありました。
*   **`touch`コマンド**: ファイルのアクセス時刻と更新時刻を現在時刻に更新するUNIXコマンドです。ファイルが存在しない場合は新しく作成します。この修正では、既存のファイルのタイムスタンプを更新することで、`make`にそのファイルが最新であると「錯覚」させるために使われています。

## 技術的詳細

このコミットの核心は、`src/cmd/gc/Makefile`内の`sysimport.c`ターゲットのビルドルールに条件分岐を導入した点です。

変更前は、`sysimport.c`のビルドルールは以下のようでした。

```makefile
sysimport.c:	sys.go unsafe.go mksys.c
	gcc -o mksys mksys.c
	6g sys.go
	6g unsafe.go
	./mksys sys >_sysimport.c &&\
		./mksys unsafe >>_sysimport.c &&\
		mv _sysimport.c sysimport.c
```

これは、`sysimport.c`が`sys.go`、`unsafe.go`、`mksys.c`のいずれかよりも古い場合、常に以下のステップを実行します。
1.  `mksys.c`を`gcc`でコンパイルして`mksys`実行ファイルを生成。
2.  `sys.go`と`unsafe.go`を`6g`でコンパイル。
3.  `mksys`を実行して`_sysimport.c`を生成し、最終的に`sysimport.c`にリネーム。

問題は、Perforceの新規チェックアウト時に`sysimport.c`が古いタイムスタンプで取得され、かつ`6g`コンパイラがまだビルドされていない（またはパスが通っていない）状況で`make`が実行されると、ステップ2の`6g sys.go`でエラーが発生し、ビルドが停止してしまうことでした。

変更後は、以下の条件分岐が追加されました。

```makefile
sysimport.c:	sys.go unsafe.go mksys.c
	if test -x $(BIN)/6g; then \
		# ... (元のビルドステップ) ...
	elif test -f sysimport.c; then \
		touch sysimport.c; \
	fi
```

この変更により、`sysimport.c`のビルドがトリガーされた際に、まず以下のチェックが行われます。

1.  **`if test -x $(BIN)/6g; then ...`**:
    *   `test -x $(BIN)/6g`は、`$(BIN)`ディレクトリ（Goのバイナリが配置される場所）に`6g`という実行可能ファイルが存在するかどうかをチェックします。
    *   もし`6g`が**存在し、実行可能であれば**、通常の`sysimport.c`のビルドプロセス（`gcc`で`mksys`をビルドし、`6g`でGoファイルをコンパイルし、`mksys`で`sysimport.c`を生成する一連のステップ）が実行されます。これは、`6g`が利用可能であるため、`sysimport.c`を正しく再生成できる状況です。

2.  **`elif test -f sysimport.c; then touch sysimport.c; fi`**:
    *   もし上記の`if`条件（`6g`が存在し、実行可能である）が**偽**であった場合、次にこの`elif`条件が評価されます。
    *   `test -f sysimport.c`は、現在のディレクトリに`sysimport.c`というファイルが**存在するかどうか**をチェックします。
    *   もし`sysimport.c`が**存在すれば**、`touch sysimport.c`コマンドが実行されます。このコマンドは`sysimport.c`の最終更新時刻を現在時刻に更新します。これにより、`make`は`sysimport.c`がその依存関係よりも新しいと判断し、それ以上の再ビルドを試みなくなります。

このロジックにより、`6g`がまだ利用できない（ブートストラップの初期段階や新規チェックアウト直後でまだコンパイラがビルドされていない）状況でも、もし`sysimport.c`が既に存在していれば、不必要なビルドエラーを回避し、ビルドプロセスを続行させることができます。`sysimport.c`のタイムスタンプを更新することで、`make`は一時的にこのファイルが最新であるとみなし、後続のビルドステップ（例えば`6g`自体のビルド）に進むことが可能になります。`6g`がビルドされた後、再度`make`が実行されれば、必要に応じて`sysimport.c`は正しく再生成されることになります。

## コアとなるコードの解説

変更された`Makefile`のルールは、Go言語のブートストラップビルドプロセスにおける堅牢性を高めるためのものです。

```makefile
sysimport.c:	sys.go unsafe.go mksys.c
	if test -x $(BIN)/6g; then \
		gcc -o mksys mksys.c; \
		6g sys.go; \
		6g unsafe.go; \
		./mksys sys >_sysimport.c && \
		./mksys unsafe >>_sysimport.c && \
		mv _sysimport.c sysimport.c; \
	elif test -f sysimport.c; then \
		touch sysimport.c; \
	fi
```

*   **`sysimport.c: sys.go unsafe.go mksys.c`**: これは`make`の依存関係定義です。`sysimport.c`は`sys.go`、`unsafe.go`、`mksys.c`に依存していることを示します。これらのいずれかのファイルが`sysimport.c`よりも新しい場合、このルールが実行されます。
*   **`if test -x $(BIN)/6g; then \ ... fi`**:
    *   `test -x $(BIN)/6g`: `$(BIN)`はGoのツールチェーンバイナリが置かれるディレクトリ（例: `$GOROOT/bin`）を指します。この条件は、Goコンパイラ`6g`が既にビルドされ、実行可能であるかをチェックします。
    *   もし`6g`が利用可能であれば、その後のブロックが実行されます。これは、`mksys`をCコンパイラ（`gcc`）でビルドし、`6g`で`sys.go`と`unsafe.go`をコンパイルし、最後に`mksys`を使って`sysimport.c`を生成するという、`sysimport.c`を完全に再ビルドする標準的な手順です。
*   **`elif test -f sysimport.c; then \ touch sysimport.c; \ fi`**:
    *   `if`条件が偽（つまり`6g`がまだ利用できない）の場合にこの`elif`ブロックが評価されます。
    *   `test -f sysimport.c`: `sysimport.c`ファイルが現在のディレクトリに存在するかどうかをチェックします。これは、Perforceなどからソースコードをチェックアウトした際に、`sysimport.c`自体は存在しているが、そのタイムスタンプが古いという状況を想定しています。
    *   `touch sysimport.c`: もし`sysimport.c`が存在すれば、このコマンドが実行され、ファイルの最終更新時刻が現在時刻に更新されます。これにより、`make`は`sysimport.c`が最新であると判断し、このビルドステップをスキップします。これは、`6g`がまだ利用できないために`sysimport.c`を正しく再ビルドできない状況で、ビルドプロセスがエラーで停止するのを防ぐための「一時的な回避策」です。ビルドの次の段階で`6g`がビルドされ、その後の`make`実行時に`sysimport.c`が本当に古ければ、正しく再ビルドされることが期待されます。

この修正は、Goの初期のビルドシステムが、まだ完全に自己ホストされていない（つまり、Goコンパイラ自体がGoで書かれていない）段階でのブートストラップの課題に対処するために重要でした。
